§3 · What we propose — refactor

One job per file

🧱 Layout

Column widths, padding, alignment — where "white space" gets fixed by design

📜 Scrolling

How the long list moves and remembers its place

⏳ Loading

When to show or hide the loading overlay

🃏 Cards

How each card displays inside a cell

🗂 Grouping & rarities

Expand / collapse, rarity badges

📌 Sticky elements

Row numbers, back-to-top buttons

🔀 Built beside the old version, behind a switch. When ready, we flip the switch. If anything breaks — flip it back instantly, no emergency deploy.
Setiap file punya satu tugas jelas. Ada file khusus Layout — di sinilah nanti masalah white-space diperbaiki lewat desain. Lalu ada Scrolling, Loading, Cards, Grouping, dan Sticky elements. Yang paling penting: semua dibangun DI SAMPING versi lama, di balik sebuah saklar. Kalau versi baru siap, kita nyalakan. Kalau ada yang rusak, kita matikan lagi seketika — tanpa deploy darurat, tanpa drama.
§4 · Why refactor — three concrete payoffs

Organized, faster to debug, easier to extend

1 · Organized, not gigantic

Understand one ~300-line file in ~30 min vs. ~12 hours reading the giant file today.

2 · Faster root cause

A "white space" report goes straight to the Layout file. First-attempt fix rate: ~10% → ~70%.

3 · Easier features & debugging

Change one file, leave others untouched. Each file has its own small tests — we know what broke before the user does.

Kenapa ini berharga? Tiga hal konkret. Satu, kodenya jadi rapi — developer bisa paham satu file 300 baris dalam 30 menit, bukan 12 jam membaca file raksasa. Dua, mencari penyebab bug jadi cepat — laporan white-space langsung menuju file Layout. Kami perkirakan tingkat berhasil sekali coba naik dari 10 persen menjadi 70 persen. Tiga, menambah fitur dan debugging jadi lebih mudah, karena tiap file punya tesnya sendiri.

§7 · The honest framing

When is each side right?

Patch — if you believe the next attempt has a 90% chance

Then refactoring is over-engineering. We should just patch it.

Refactor — 22 prior failures suggest the opposite

The next patch is no more likely than the last. The structure of the code is the obstacle, not the developer's skill.

🔍 The indirect benefit: even if the refactor doesn't directly find the root cause, it makes it possible to find. The search space shrinks from 4,778 lines → ~300.
Saya ingin jujur soal kapan masing-masing pilihan benar. Kalau Anda yakin percobaan ke-24 punya peluang 90 persen berhasil, maka jangan refactor — tambal saja, itu over-engineering. Tapi 22 kegagalan sebelumnya menunjukkan sebaliknya. Yang jadi penghalang itu STRUKTUR kodenya, bukan kemampuan developer-nya. Dan manfaat tidak langsungnya: walau refactor sendiri tidak langsung menemukan akar masalah, ia membuat akar masalah itu MUNGKIN ditemukan — ruang pencarian menyusut dari 4.778 baris jadi sekitar 300.
§7a · Suspected causes of the white-space issue

A bounded set of 6 suspects — each isolatable after refactor

#Candidate causeWhy it shows as white spaceOwning module
1Virtualization window lagWindow advances faster than rows mount → blank gap until catch-upScrolling
2Wrong / estimated row heightsSpacer math off → gaps open between rowsLayout
3Async image decodeArea paints before images decode → cells flash emptyCards + Loading
4Overlay hides too earlyLifts on data arrival, not render-complete → unpainted rows showLoading
5Scroll-restore jumpPosition restored before layout settles → momentary blankScrolling
6Sticky / offset miscalcGeometry off by a few px under fast scrollSticky elements
⚠️ Hypotheses to confirm in the Week-1 spike — not findings. The likeliest culprits (#1, #2, #5) all live in scrolling + layout, today interleaved with everything else. A fix for #1 routinely triggers a regression that looks like #4 — isolating them is what makes Round 24 different.
Ini menjawab pertanyaan soal penyebab. Masalah white-space sebenarnya punya enam tersangka — mulai dari keterlambatan rendering saat scroll cepat, salah hitung tinggi baris, decode gambar yang asinkron, sampai overlay yang hilang terlalu cepat. Saya tegaskan: ini masih HIPOTESIS yang akan kami buktikan di spike Minggu 1, bukan kesimpulan final. Tapi perhatikan — tersangka terkuat, nomor 1, 2, dan 5, semuanya ada di scrolling dan layout, yang hari ini tercampur dengan segalanya. Itulah kenapa kita tidak bisa mengisolasinya sekarang, dan itu yang membuat Ronde 24 berbeda.

§11–12 · Safety levers & risks

Every safety lever is manual — the team stays in control

🧯 When something goes wrong

Prod regression → flip the switch off, users see the old version instantly. No 2 AM page.
Parity bug in W5 → delay cutover 2–3 days, don't ship until clean.
Mid-refactor feature request → backlog it. Scope is not bent.

⚠️ Top risks, handled

Hidden complexity → Week-1 spike.
Slippage → Week-7 buffer pre-approved.
Scope creep → strict "parity only".
Old code lingers → deletion in W6 is a gate, not optional.

🚦 Protecting against scope creep is the PM's primary job for these 6 weeks — it's the most common cause of refactor-project failure.
Soal keamanan, semua tuas pengaman kita buat manual supaya tim tetap pegang kendali. Kalau ada regresi di produksi, engineer cukup matikan saklar, user langsung lihat versi lama, tidak ada yang dibangunkan jam 2 pagi. Kalau QA menemukan bug paritas di Minggu 5, cutover ditunda 2 sampai 3 hari, kita tidak rilis sampai bersih. Kalau ada permintaan fitur di tengah jalan, kita catat ke backlog — scope tidak ditekuk. Menjaga scope inilah tugas utama PM, karena scope creep adalah penyebab nomor satu kegagalan proyek refactor.
