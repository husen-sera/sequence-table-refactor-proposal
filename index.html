Straight answersThe four questions you're likely asking

Will this actually solve the bug — and how long will it take?

Yes — and the surest path is to fix the structure first. The refactor (~6 weeks manual, ~4 weeks with AI) sets up the white-space fix as a small, isolated follow-up in Week 7 — about 1.5 days of work with ~70% odds of fixing it on the first attempt, versus ~10% today. If your priority is instead a guaranteed same-day patch, that's a valid goal — we can choose that plan together. This one optimizes for the durable win.

What's actually causing the white-space issue?

It comes down to a bounded set of six likely suspects — things like the list rendering lagging behind a fast scroll, or images decoding a beat too late. Today these are tangled together in one file, so no one can isolate which one fires. After the refactor, each suspect lives in its own small file and can be tested — and fixed — on its own. (Full list further down.)

Can other development keep moving while this happens?

Mostly, yes. Everything outside the sequence-table view keeps going — about 70% of normal frontend output and 100% of backend work continue. The only work we pause is new features inside this specific view, which we queue to Week 7 so they land on the clean foundation instead of needing rework.

We use AI — how much faster does that make it?

AI compresses the reading and building phases significantly — roughly 6 weeks down to 4. The validation phases (quality testing, the production safety period) are bound by the calendar, not typing speed, so they stay the same. Importantly, the 4-week figure assumes the same testing rigor as the 6-week plan — AI speeds up writing the code, it doesn't lower the bar for checking it.

What we proposeSplit it into 5–6 small, focused files

Refactoring means rewriting code without changing what the user sees — making it easier for the next person to understand and safely modify. We split the same 4,778 lines so each file does one clear job:

🧱 Layout

Column widths, padding, alignment — where the "white space" gets fixed by design.

📜 Scrolling

How the long list moves and remembers its place.

⏳ Loading

When to show or hide the loading overlay.

🃏 Cards

How each card displays inside a cell.

🗂 Grouping & rarities

Expand / collapse, rarity badges.

📌 Sticky elements

Row numbers, back-to-top buttons.

🔄 The new version is built beside the old one, behind a switch. When it's ready, we flip the switch. If anything looks off, we flip it back — instantly, with no emergency deploy.

Staying in controlWhat happens if something slips

🧯 If a user hits a problem in production

It's a one-click rollback, not a 2 AM emergency. We flip the switch off, users instantly see the proven old version, and we investigate calmly the next working day.

🧪 If testing finds a mismatch in Week 5

The date moves, never the quality bar. Cutover shifts 2–3 days while we fix and re-test — we ship only once the two versions match.

💡 If a new feature request comes mid-way

We capture it and schedule it for after Week 6, so it lands on the better foundation. Protecting this queue is the project manager's main job.

📅 If the timeline slips

It stays visible. A pre-approved 7th-week buffer absorbs surprises, and you hear about any change within 24 hours — no silent slippage.

ScopeWhat's in, and what stays safely out

The simpler 1-table and 2-table views · search, login, settings, history · the backend, API, and database · the in-progress backend cutover. Zero risk to any of these.
